## Commands

```bash
uv sync                                                          # install dependencies
uv run alembic upgrade head                                      # apply migrations (required before first run)
uv run uvicorn --app-dir src evalops_dashboard.main:app --reload # run the API (http://127.0.0.1:8000/docs)
uv run ruff format .                                              # format
uv run ruff check .                                               # lint
uv run pytest                                                     # run all tests
uv run pytest tests/test_comparison.py                            # run a single test file
uv run pytest tests/test_comparison.py::test_name -v               # run a single test
EVALOPS_DATABASE_URL="postgresql+psycopg://..." uv run pytest postgres_smoke_test/ -v # Postgres smoke test (needs a live Postgres; not part of the default suite)
```

CI (`.github/workflows/ci.yml`) has two jobs: `test` runs `ruff format --check`, `ruff check`, `alembic upgrade head`, then `pytest` against SQLite — run all four before finishing a change. `postgres-smoke` runs migrations and `postgres_smoke_test/` against a real `postgres:16` service container to verify PostgreSQL compatibility.

Database URL is configurable via `EVALOPS_DATABASE_URL` (defaults to `sqlite:///./evalops.db`); CI's `test` job uses a separate `ci_evalops.db`. The app also runs unchanged against PostgreSQL (`postgresql+psycopg://...`) — see the README's PostgreSQL Support section.

## Architecture

FastAPI + SQLModel app under `src/evalops_dashboard/`, with routers mounted onto `main.py`:

- `models.py` — all SQLModel table models plus their `*Create`/`*Read` DTOs live in one file. Table models (`Prompt`, `ModelResponse`, `Rubric`, `RubricCriterion`, `Evaluation`, `CriterionScore`) and their API-facing schemas are colocated here; there's no separate schemas package.
- `database.py` — single SQLAlchemy `engine`, built from `EVALOPS_DATABASE_URL`. Uses `StaticPool` only for the in-memory `sqlite://` test case, and enables SQLite foreign keys via a `connect` event listener; both are gated behind a `sqlite` URL check, so the same engine setup runs unchanged against PostgreSQL (verified by the `postgres-smoke` CI job).
- `main.py` — app instance, lifespan hook that seeds demo data on startup via `seed.py` (the app never creates/alters tables at runtime — schema changes only happen through Alembic), and the `/health`, `/analytics/summary`, `/prompts`, `/responses` routes directly. Also registers the `DashboardAuthRequired` exception handler (redirects unauthenticated dashboard visits to `/login`).
- `routers/` — `evaluations.py`, `rubrics.py`, `comparisons.py`, `analytics.py` hold the remaining route groups, each taking a `SessionDep` (`Annotated[Session, Depends(get_session)]`). Every router now also takes `CurrentUser` (`Annotated[User, Depends(get_current_user)]`, JSON routes) or `CurrentDashboardUser` (dashboard routes) — both defined in `auth.py` — except `GET /health`, `GET /login`, `POST /login`, and `POST /auth/login`, which stay public by design.
- `scoring.py` — pure functions for weighted overall-score and pass/fail calculation used by the evaluations router; formula and pass rules are documented in the README's "Evaluation Scoring" section.
- `comparison.py` — pure aggregation logic for the `/prompts/{id}/comparison` endpoint (ranking, tie-breakers, `comparison_ready`/`winner_response_id` derivation). `routers/comparisons.py` loads the ORM rows, converts them into this module's plain dataclasses, calls `build_comparison_summary`, then maps the result back onto the `PromptComparisonRead` response model. When changing comparison behavior, the ranking/aggregation logic belongs in `comparison.py`, not the router.
- `criteria_analytics.py` — pure aggregation for `GET /analytics/by-criterion` (`routers/analytics.py`): groups `CriterionScore` rows by `RubricCriterion.name` (a string, not `criterion_id` — the same criterion name can exist as different rows in different rubrics, since `RubricCriterion`'s uniqueness is scoped to one rubric, and grouping by name across all of them is what makes this cross-rubric rather than per-rubric), then by `model_name` within each. This endpoint lives in its own router file rather than directly in `main.py` (unlike `/analytics/summary`) specifically so `routers/dashboard.py`'s `/dashboard/analytics` page can import and call `get_criteria_analytics` directly as a plain function, the same way it already calls `compare_prompt_responses` from `routers/comparisons.py` — `main.py` imports `routers/dashboard.py`, so a route defined in `main.py` itself can't be imported back by `dashboard.py` without a circular import.

This split (pure calculation module + thin router that does ORM I/O and DTO mapping) is the pattern to follow for new score- or ranking-related features — see `scoring.py`/`evaluations.py`, `comparison.py`/`comparisons.py`, `criteria_analytics.py`/`routers/analytics.py`, and `cost.py`/`routers/pricing.py` (calculation) + `main.py`'s `create_model_response` (the calling site) as four examples.

- `cost.py` — pure `calculate_cost(input_tokens, output_tokens, input_price_per_1k_tokens, output_price_per_1k_tokens)`. `routers/pricing.py` is the CRUD router for the `ModelPricing` catalog (create/list/detail, mirrors `rubrics.py`'s shape). `main.py`'s `create_model_response` looks up a `ModelPricing` row by `(provider, model_name)` and calls `calculate_cost` when both `input_tokens`/`output_tokens` are provided — an unmatched provider/model or missing tokens just leaves `cost_usd` unset, not an error. `cost_usd` deliberately lives only on the `ModelResponse` table model, not on `ModelResponseBase` (which `ModelResponseCreate` inherits from) — this keeps it out of client-submittable input, the same way `Evaluation.overall_score`/`passed` are excluded from `EvaluationCreate`.

- `llm_judge.py` — the app's first outbound third-party network calls: builds a scoring prompt and a per-criterion JSON tool schema (pure functions, testable with zero mocking), calls a configurable provider to get structurally-guaranteed per-criterion scores, and produces an `EvaluationCreate` that `routers/evaluations.py`'s `create_auto_evaluation` (`POST /evaluations/auto`) hands to the existing `create_evaluation_from_payload` unchanged — this module's only job is producing valid input for that path, not duplicating its validation/scoring/persist logic. Two providers implement the `JudgeClient` protocol (`evaluate()` + an `evaluator_name` used to tag the resulting `Evaluation`): `AnthropicJudgeClient` (default; requires `ANTHROPIC_API_KEY`, raises a clear `503` if missing, checked lazily on first use, never at import time; model is `ANTHROPIC_JUDGE_MODEL`, falling back to `claude-haiku-4-5-20251001`; supports forcing the tool call via `tool_choice`) and `OllamaJudgeClient` (opt-in via `LLM_JUDGE_PROVIDER=ollama`; talks to `OLLAMA_HOST`, falling back to `http://127.0.0.1:11434`; model is `OLLAMA_JUDGE_MODEL`, falling back to `qwen2.5:1.5b`; no forced-tool-choice equivalent, so a declined tool call is a normal, handled `502`). Both providers extract a raw `dict` from their own SDK's response shape, then hand it to one shared, provider-agnostic validator, `build_judge_result_from_tool_input` — the only thing that differs between `parse_anthropic_tool_response`/`parse_ollama_tool_response` is that extraction step. `get_judge_client()` picks the provider from `LLM_JUDGE_PROVIDER` (default `"anthropic"`). Like `auth.py`, this is a documented deviation from the pure-module pattern: both clients do real I/O, exposed as one FastAPI dependency (`get_judge_client`) specifically so tests can override it via `app.dependency_overrides` the same way `auth.py`'s dependencies are overridden — no real network calls happen in `uv run pytest`. `postgres_smoke_test/`'s pattern (a live-infra test outside `tests/`, only run explicitly or in its own CI job) is reused twice more here: `ollama_smoke_test/` gets real end-to-end CI coverage against a real `ollama/ollama` service container (Ollama is free, so unlike Anthropic this doesn't need a secret) — the Anthropic path has no live-success test for the same reason Postgres originally had no local Docker: nothing to run it against in this environment. `JudgeResult` also carries `input_tokens`/`output_tokens`/`model`, read directly from the real API response (Anthropic's `message.usage`/`message.model`; Ollama's `response.prompt_eval_count`/`eval_count`/`response.model` — confirmed via direct inspection of both installed SDKs before writing this, not assumed) and threaded through the shared `build_judge_result_from_tool_input` as plain pass-through values. `build_auto_evaluation_create(model_response, rubric, judge_result, evaluator_name)` is a pure mapping function — the network call (`judge_client.evaluate(...)`) happens in `routers/evaluations.py`'s `create_auto_evaluation`, not inside this function, specifically so the router can see the raw `JudgeResult`'s token counts to look up `ModelPricing` and calculate `judge_cost_usd` via `cost.py`'s existing `calculate_cost` — no new pure-calculation code for this. `judge_input_tokens`/`judge_output_tokens`/`judge_model`/`judge_cost_usd` live only on `Evaluation` (never `EvaluationBase`/`EvaluationCreate`), the same "server-controlled results" placement `ModelResponse.cost_usd` already uses — `create_evaluation_from_payload` takes them as optional keyword params so the human `POST /evaluations` call site simply never passes them. `seed.py`'s `ensure_seed_judge_pricing` is a separate function from `ensure_seed_pricing` on purpose: that function's idempotency check is scoped to `ModelPricing.provider == SEED_PROVIDER`, so extending its list with a different provider would silently fail to detect an existing row and hit `ModelPricing`'s unique constraint on every restart.

- `routers/evaluations.py` also exposes `create_evaluation_from_payload` (the persist-one-evaluation core — validation, scoring, and the transactional persist block — extracted from the `POST /evaluations` route so both that route and the CSV import loop share exactly one code path) and `build_csv_columns` (rubric-criteria-derived CSV header, shared by the CSV export and import routes). New evaluation-persistence logic should extend `create_evaluation_from_payload` rather than duplicating the transaction/validation flow; new CSV column logic should extend `build_csv_columns` rather than hardcoding headers in two places.

- `routers/dashboard.py` — server-rendered HTML routes under `/dashboard` (Jinja2 templates in `templates/`, static assets in `static/`, mounted via `StaticFiles` at `/static` in `main.py`). This one has no pure-calculation counterpart: it's read-only presentation over existing data, and reuses `build_rubric_response` (from `rubrics.py`), `build_evaluation_responses` (from `evaluations.py`), and `compare_prompt_responses` (from `comparisons.py`, called directly rather than re-deriving its aggregation) rather than introducing new business logic. Prompt/response queries have no existing builder either, so those are inlined directly in the route functions. `build_applicable_rubrics` and `build_chart_rows` (with the `BarChartRow`/`CriterionChart`/`ChartData` dataclasses) also live here — the latter is presentation reshaping (rounding, percentage-of-max math for bar widths) on top of `compare_prompt_responses`'s output, not aggregation, which is why it stays out of `comparison.py`. Charts themselves are static server-rendered bars (a `bar_chart` Jinja macro in `templates/partials/bar_chart.html`) with no client-side JavaScript, consistent with the rest of the dashboard.

- `auth.py` — a fourth example of the pure-module + thin-router pattern, with one documented deviation: its session functions (`create_session`, `resolve_session_token`, `invalidate_session`) do their own DB I/O rather than staying pure, because that I/O is needed identically by three call sites (JSON login/logout, dashboard form login/logout, and the `get_current_user`/`get_current_dashboard_user` dependencies) — putting it in one place avoids duplicating the same queries three times. Also holds `hash_password`/`verify_password` (bcrypt), `authenticate_user` (deliberately returns `User | None` without distinguishing "no such user" from "wrong password," to avoid username enumeration — callers must show one generic error), and the `DashboardAuthRequired` exception used by the redirect-to-login mechanism. `routers/auth.py` has three `APIRouter`s in one file (justified by URL-prefix constraints): `/auth/login`+`/auth/logout` (JSON), unprefixed `POST /users` (admin-only), and unprefixed `GET/POST /login`+`POST /logout` (dashboard forms) — the dashboard form routes call the same `authenticate_user` core as the JSON login route rather than duplicating credential-check logic. Sessions are server-side rows (`AuthSession` table, opaque `secrets.token_urlsafe` tokens), not JWT — see the README's Authentication section for why. Note: `AuthSession.expires_at` needs its `tzinfo` re-attached after reading back from SQLite (see the comment in `resolve_session_token`) — SQLite drops timezone info on datetime round-trip, which only matters here because this is the first place the codebase compares a stored datetime rather than just displaying it. Login rate limiting (`is_login_rate_limited`, `record_login_attempt_result`) lives in this same file and follows the same real-I/O deviation: a `LoginAttempt` row per failed attempt (keyed on the raw username string, not a `User` FK, so nonexistent usernames are limited identically to real ones — same anti-enumeration principle as `authenticate_user`), a successful login clears that username's rows, and the window check compares entirely inside the SQL `WHERE` clause rather than re-comparing Python `datetime` objects after a round-trip, so it doesn't hit the SQLite tzinfo bug above. Both `login()` and `login_submit()` in `routers/auth.py` call it the same way, before `authenticate_user` — `authenticate_user` itself is unmodified. Role-based access control is a `role: UserRole` (`"admin"`/`"member"`, defaults to `"member"`) field on `UserBase` — shared by `User`, `UserCreate`, and `UserRead` — and one dependency, `get_current_admin_user`/`CurrentAdminUser`, composed on top of `get_current_user`/`CurrentUser` rather than duplicating auth logic. `role` is deliberately forced to a plain `sa.String(length=20)` column (`sa_type=` on the `Field`) instead of the native Postgres `ENUM` type SQLModel would otherwise generate for a `str` `Enum` field — confirmed via direct DDL inspection — to avoid `CREATE TYPE`/`DROP TYPE` lifecycle complexity in the migration; Pydantic/FastAPI still fully validates the field at the request boundary regardless of the underlying column type. `POST /users` is currently the only route `CurrentAdminUser` gates — everything else stays open to any authenticated user, a deliberately narrow first cut rather than a full permission matrix.

### Migrations

Schema changes go through Alembic (`alembic/versions/`), not app startup. `alembic/env.py` imports `evalops_dashboard.models` for autogenerate metadata and reads `EVALOPS_DATABASE_URL` the same way the app does. Breaking migrations (see the `20260710_0002` migration) are acceptable pre-release but must be called out in the README's Database Migrations section, including what data they reset vs. preserve.
