## Milestone 9 — Model/Provider Metadata and Cost Tracking
- **Status:** Done
- **Merged:** 2026-07-10T23:40:36Z, squash commit [`a82ed97`](https://github.com/yaja95/evalops-dashboard/commit/a82ed9737235f61915860f3d62d0fe54930d5456) on `main`, PR [#7](https://github.com/yaja95/evalops-dashboard/pull/7). Verified via `gh pr list --state merged` and `git log`/`git ls-remote` against `origin/main` directly.
- **Post-merge checks re-run against `main`:** `ruff format --check .`, `ruff check .`, `alembic upgrade head`, `pytest` — all pass, 121/121 tests.
- **Shipped:** `model-pricing` catalog (`POST/GET /model-pricing`, `GET /model-pricing/{id}`) plus server-calculated `cost_usd` on model responses. Clients submit `provider`/`input_tokens`/`output_tokens` as raw facts; the server calculates cost from a matching pricing entry — clients cannot submit `cost_usd` directly (422 on attempt), following the same server-controlled-results principle already applied to evaluation scoring. `cost_usd` deliberately lives only on the `ModelResponse` table model, not the shared `ModelResponseBase` that `ModelResponseCreate` inherits from, so the client-input boundary is structural, not just a runtime check. Unmatched provider/model or missing tokens leaves `cost_usd` unset — not an error. `model_name` was deliberately left untouched (not split into provider+model), so `comparison.py`/`comparisons.py` and existing templates needed zero changes. First schema migration since Milestone 4 (non-breaking: four nullable columns + one new table), tested in both directions against a scratch database before being wired into anything else.
- **No new bug class this time** — implementation matched the plan cleanly, verified against hand-calculated cost values before writing automated tests.

## Milestone 10 — Authentication for Internal Team Usage
- **Status:** Done
- **Merged:** 2026-07-11T03:00:47Z, squash commit [`c3d32df`](https://github.com/yaja95/evalops-dashboard/commit/c3d32df8372acb537c6aec78555c1eec493644d4) on `main`, PR [#8](https://github.com/yaja95/evalops-dashboard/pull/8). Verified via `gh pr list --state merged` and `git log`/`git ls-remote` against `origin/main` directly.
- **Post-merge checks re-run against `main`:** `ruff format --check .`, `ruff check .`, `alembic upgrade head`, `pytest` — all pass, 133/133 tests. Real browser login flow re-verified against the merged code (not just pre-merge).
- **Shipped:** Session-based auth (`AuthSession` table, opaque `secrets.token_urlsafe` tokens — not JWT, to keep the DB as the single source of truth for all state and make logout actually revoke access) required on all 28 pre-existing routes plus new auth routes, except `/health`, the login page, and static assets. `POST /auth/login` sets an `HttpOnly`/`SameSite=Lax` cookie for the dashboard and returns the token for `Authorization: Bearer` API use. No self-registration — seed `demo` user (password via `SEED_USER_PASSWORD` env var, documented fallback) plus authenticated-only `POST /users`. `password_hash` lives only on the `User` table model, never the shared base `UserCreate` inherits from — same structural boundary as `cost_usd` on `ModelResponse`. Existing 121 tests needed **zero individual changes**, via a `conftest.py` autouse fixture using FastAPI's `dependency_overrides`; confirmed empirically, not assumed.
- **Two bugs found and fixed during implementation, not planning:** (1) SQLite drops timezone info on datetime round-trip — `resolve_session_token`'s expiry check crashed comparing a naive DB value against an aware "now" until UTC was re-attached. This is the first place the codebase ever compared a stored datetime rather than just displaying one, so the gap was latent since Milestone 0. Worth remembering for any future code that compares (not just displays) a `datetime` field. (2) A design-agent exploration subagent falsely claimed a pre-existing syntax error in `evaluations.py` and had autonomously spawned a task-suggestion chip based on that false premise — the "error" was actually valid Python 3.14 syntax (PEP 758, unparenthesized multi-exception `except` clauses); the agent had checked with an older Python interpreter that predates PEP 758. Dismissed the stale chip after independently verifying with `uv run python` (this project's real pinned interpreter). Worth remembering: always verify a sub-agent's environment-dependent claims (syntax errors, version-gated behavior) against the project's actual pinned toolchain before acting on them.
