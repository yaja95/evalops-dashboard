- **Status:** Done. Closes a gap the README named explicitly as a deliberate scope cut since Milestone 10 ("Rate limiting. Not built in this version..."). Two backlog items remain open, unmilestoned: cross-rubric analytics, role-based access control.
- **Merged:** 2026-07-12T00:48:14Z, squash commit [`d14163e`](https://github.com/yaja95/evalops-dashboard/commit/d14163ea083bfc2a46aa1b8b0beade319ba25b1b) on `main`, PR [#11](https://github.com/yaja95/evalops-dashboard/pull/11). - **Shipped:** After 5 failed login attempts for a username within 15 minutes, both `POST /auth/login` (JSON) and the dashboard `POST /login` form return `429` for further attempts against that username — even with the correct password — until the window passes; a successful login clears the counter. New `LoginAttempt` table (one row per failed attempt, `username` as a raw indexed string rather than a `User` FK) backs this, matching the DB-as-single-source-of-truth design `AuthSession` already established, rather than an in-memory counter that wouldn't survive a restart or a multi-worker deployment. Keyed by username, not IP — deliberately, to match the exact roadmap wording, with the resulting denial-of-service-via-lockout tradeoff named explicitly in the README rather than hidden. Attempts are recorded identically whether or not the username exists, preserving `authenticate_user`'s existing anti-enumeration guarantee. `authenticate_user` itself was not modified — the two new functions (`is_login_rate_limited`, `record_login_attempt_result`) wrap around it from both call sites.
- **No new bug, but the rate-limit window check was specifically designed and tested to avoid a known bug class already documented in this file:** `resolve_session_token`'s SQLite tzinfo-on-read-back issue (Milestone 10) only bites when two Python `datetime` objects are compared after one has round-tripped through SQLite. `is_login_rate_limited` avoids that entirely by comparing inside the SQL `WHERE` clause (DB column vs. a bound parameter) rather than fetching and re-comparing in Python — confirmed empirically by the passing `test_login_rate_limit_ignores_attempts_outside_window` test (a directly-backdated row correctly falls outside the window) rather than just assumed from the design.
- **Verification went a step further than "does it pass":** beyond the automated suite, ran the app locally against a scratch DB and hit the real endpoint via curl 6 times (5 wrong passwords, then the correct one), observing the real `401`×5 → `429` sequence firsthand before considering the feature done — the same bar as the Ollama live-verification win in Milestone 12, applied here because it was actually possible to run locally this time (no external API/service needed).

- **Shipped:** `POST /users` (creating a new account) now requires an admin — the *only* route this milestone gates, confirmed with the user up front rather than assumed, since the roadmap line ("admin vs. member permissions") didn't specify scope on its own. Everything else stays open to any authenticated member, unchanged. New `role: UserRole` field (`"admin"`/`"member"`, defaults to `"member"`) lives on `UserBase`, so `User`, `UserCreate`, and `UserRead` all share it — deliberately different from the "never client-submittable" pattern (`cost_usd`, `overall_score`): since `POST /users` is now admin-only, it's correct for an admin's creation payload to set the new account's role, not a privilege-escalation hole. `get_current_admin_user`/`CurrentAdminUser` composes on top of the existing `get_current_user` rather than duplicating auth logic. Seeded `demo` user promoted to `admin` both in `seed.py` (fresh installs) and via a migration data-fix (`UPDATE "user" SET role = 'admin' WHERE username = 'demo'`, for pre-existing databases) — verified directly by seeding a pre-migration `demo` + a second user, running the upgrade, and confirming only `demo` got promoted.
- **One real design decision worth remembering:** a bare `role: UserRole = Field(default=UserRole.MEMBER)` would have made SQLAlchemy generate a *native Postgres `ENUM` type* for a `str`-`Enum`-typed field (confirmed via direct DDL inspection before writing the migration) — which would have meant real `CREATE TYPE`/`DROP TYPE` lifecycle complexity in the migration, tested against this repo's actual `postgres-smoke` CI job, not just a SQLite-only sanity check. Forced a plain `VARCHAR(20)` instead via `sa_type=sa.String(length=20)` on the `Field`, confirmed empirically that Pydantic/FastAPI still fully validates the field at the real client-facing request boundary regardless (a bad role value gets a clean `422`), and that only irrelevant, already-trusted raw table construction skips that validation. Worth remembering for any future enum-like column on this project: check what SQLAlchemy would generate by default before writing the migration, don't assume.
- **Verification note:** two pre-existing tests (`test_create_user_creates_a_user_who_can_then_log_in`, `test_create_user_rejects_duplicate_username`) needed updating, not just new tests added — both had been logging in as a freshly-created non-admin user and successfully calling `POST /users`, which is exactly the behavior this milestone removes.
